# 14 — 机器够不到的业务真相：判据退化为封条

Type: grilling
Status: resolved
Blocked by: 10

## Question

当业务正确性**机器够不到**时，判据退化成**封条**——查「证据 JSON 在」+「源码 SHA-256 指纹匹配」。这类工作在 taskloop 里的合法形态是什么？

**出票来源**：owner 2026-07-16 在票 10 grilling 中提出——「Java 中业务比较复杂，可能单个测得没有问题，但是串起来就有问题了。TDD 只能保证编译没有错，不能保证业务的正确性。」核实后发现**它逐字写在账本里**，且是 A1 死因的上游。详见[判据传输形态决策记录](../../../docs/decisions/2026-07-16-criterion-transport-form.md)「决议 1 的限制」节。

**硬证据**（A1 `1e123be8`，`proof_gap_acceptances[0]`，`granted_by: self`）：

> 外部 SOFARPC/DB 结果无法由只读本地判据重复探测；本次会话已实际执行两次 updateSeasonV2 与 rankingList，证据 JSON 记录请求号、前后映射和榜单结果，并用源码 SHA-256 指纹绑定当前实现；**本地 7 个聚焦测试已通过**

「本地 7 个聚焦测试已通过」= 单个测得没问题。「外部 SOFARPC/DB 无法由只读本地判据重复探测」= 串起来机器够不到。终态 `achieved` + `proof.state: provisional`。

**为何是新洞、不是既有票的重复**：

- **不是** fog 里「无机器判据的仓内工作（规划/裁决/文书）」——那类**没有客观真相**；这类**有客观真相，只是机器够不到**。
- **不是** judgmentloop——那是**品味型**交付物（人的显式验收作为终结动词，无客观对错）；这类有明确对错，人验得了，机器验不了。
- **不是**票 07/12 的信任链——那问「凭什么信这次评审/授权」；这票问「**判据该长什么样**」。（本票的 `granted_by: self` 面属 fog「人类授权的不可伪造性」，已在册，不在本票。）

**故它是第三种东西**：客观真相存在 + 机器够不到 + 人能一次性验证。票 05 决议 2「计划必须自带可执行判据」在它身上落不了地——不是判据弱（票 07 已裁），也不是判据不存在（fog 那条），而是**判据只够得着真相的影子**。

裁决问题：

1. **封条算不算判据？** 红→绿对封条证明的是「证据文件出现了」，不是「业务对了」。若算，则 `open_requirement: "unsatisfied"` 要求的红是「证据还没生成」——那是**任务开始时必然为真**的事，红见证退化成同义反复（与票 10 决议 1 诊断的「恒红无鉴别力」同形，只是这次恒红是**设计如此**）。若不算，taskloop 今天用 workloop 跑它就是形态错配。

   **活证（2026-07-16 核实，A1）**：封条读的是 `docs/test-runs/2026-07-13-field-option-team-id-stability-e2e.json` 的 `status` 字段——该文件**未提交、未被 ignore**（故在 `repoSnapshot` 内），内容为 agent 手写的 `status: "passed"` + `sourceFingerprint` 数组 + 一段自述 `notes`（「Authorized test-environment validation. The same full SeasonBgOptRequest was saved twice…」）。

   而 rev16 amend（修 `Test-Path` 语法）之后的 resume 原话是：「修正版判据**默认失败见证已记录**；**恢复真实 E2E passed 状态**并执行最终验证」——**「恢复」蕴含此前 `status` 被置为非 passed**。即：判据修好后本会立刻变绿（证据 JSON 早已 passed），红见证拿不到 → **agent 把自己写的 `status` 掰成失败以取得红见证，再掰回去**。（证据等级：agent 自述文本的蕴含，非文件版本史直接观测——该文件未进 git，无版本史可查。seq 7 的 resume 有同构表述「新判据默认失败见证已记录」。）

   **故封条上的红→绿，证明的是「agent 改了一个它自己写的字段」。** 红见证不是被绕过，是**被仪式化**——这是本票最尖的那一刀：票 10 决议 1 要求「红必须是判据亲口说的」，而封条的红确实是判据亲口说的，判据也确实跑了——**可它说的是 agent 让它说的**。**决议 1 的「可归因」到此触底**：归因到判据，判据归因到 agent 自己写的一行 JSON。
2. **封条的验收谁批？** A1 是 `granted_by: self`——人的两次手工调用只作为 reason 字符串里的自述存在。这与 judgmentloop 的「人的显式验收作为终结动词」是同一个需求吗？若是，为何 A1 走了 workloop？（形态选择today 由谁决定、开单时能否判？——参考票 08「准入式判据」的形状。）
3. **指纹绑定是不是正解？** A1 用源码 SHA-256 把「人验过的那一刻」钉在实现上。这是封条唯一有机器牙齿的部分——它防的是「验完之后又改」。它够吗？漏了什么（依赖变更、配置、数据、外部系统自己变）？
4. **代价**：若封条要成为一等形态，它与 `proof.state: provisional`/`proof_gap_accepted` 是同一个东西换名字，还是要新机制？今天 provisional 是**如实标注**（taskloop 没假装封条是判据，这一点做对了）——那缺口到底在哪？先说清缺口，再谈机制。

**落点**：待裁（可能 skills 形态选择文本 / runtime / 两者）。

**开工须先自检**：不要拿当前实现当参照系——本图已四次栽在这上面（票 02 两次、票 03 一次、票 10 一次）。先问「一个业务真相机器够不到的循环该长什么样」，再看 taskloop 有什么。

## Answer

owner 2026-07-16 确认。详见[封条形态决策记录](../../../docs/decisions/2026-07-16-unreachable-business-truth-seal.md)。

**物种定义修正**:承重事件是「一次性、不可复跑的验证行动」,执行者可为人授权下的 agent(A1 实况:人授权部署与 RPC,agent 执行并观测,人没亲手验过结果)。不可复跑是本质,人身不是。

**根问题**:声明被洗进观测通道(判据读 agent 手写的 `status` 等 verdict 字段),而红见证逐代强索是机器规则(`task-engine.mjs:192-194`,amend 换代即 hold `unsatisfied_not_witnessed`)——翻转仪式是被机器强迫的。

**四条决议**:

1. **拆焊**:封条的绑定半边(存在/schema/env/指纹)是真判据,红色词汇表 = 缺据/陈旧/**漂移**;verdict 字段是声明,**判据禁读**;红见证供给侧归机器半边。
2. **裁决通道 = proof_gap acceptance**(归位而非新造):署名如实(人亲验 `user`、agent 一次性验证 `self`),self 合法且已被 floor+强制评审正确计价——A1 全链开火,只是评审看不见未提交的收据。不需要 judgmentloop 人终结动词。
3. **指纹保留**(封条唯一的观测)+ **收据必须入 git**(买篡改可见性,不买作者真实性);覆盖缺口由 `criterion_input_coverage` 如实计价(已有)。
4. **零新机制**:`provisional` 是对封条的如实标注,机制层没有一环失职;缺口全在形态层——三件执行项进 #04(封条形态准入规则、判据写作三禁、收据入 git)。

**裁决问题四问对应**:①封条算判据但只算绑定半边;红见证向封条索要 = 制造仪式。②acceptance 通道 + 如实署名;与 judgmentloop 不是同一需求(这里有客观真相,第二双眼由强制评审提供)。③指纹是正解且是封条唯一机器牙齿,补收据入 git 使评审可见历史。④不是换名——缺口不在机制在形态,零新 runtime 机制。

**翻盘条件**:人-面身份锚出现,或形态纪律成文后伪造复发 → typed `human_verification` 一等化重问。
