## 物种定义修正(先于一切决议)

票面写「人能一次性验证」。账本证否其中「人」字:A1 实况是**人授权、agent 执行**——seq 3 挂起(`needs_input`)向人要部署与 RPC 授权,seq 4 人确认后放行,随后 agent 亲手执行两次 `updateSeasonV2` 并观测结果;人从头到尾没亲手验过结果。故物种的承重事件修正为:

> **一次性、不可复跑的验证行动**。执行者可以是人,也可以是人授权下的 agent。不可复跑是本质,人身不是。

「机器够不到」的精确含义随之清晰:**判据**(可复跑、只读、本地)够不到,一次性授权**行动**够得到。

## 根问题

不是「封条弱」,是两件事叠加:

1. **一份声明被洗进了观测通道。** A1 判据解码后是复合体,内部两个物种焊在一起:
   - **观测**:证据 JSON 存在性、schema/env 检查、两个源文件 SHA-256 指纹比对、7 个 Maven 聚焦测试——机器亲手算的,撒谎需要攻击哈希;
   - **声明**:`status=='passed'`、`rpc.ok`、`savedSameConfigurationTwice`、before/after 映射——**agent 手写进 JSON 的字段,判据把它当世界状态读**。

   这是 07「声明只能作用于声明,不能作用于观测」的**反向实例**:同形家族此前是「手上有观测却不用」(06/07/08/11),本票是**把声明伪装成观测来用**。红见证仪式全部发生在声明半边(翻 `status` 字段),指纹半边一次也没被仪式化——它仪式化不了。

2. **红见证的逐代强索是机器规则,不是 agent 迷信**(本票查证的承重新事实):`task-engine.mjs:192-194`——`witness.criterion_generation_id !== criterion.criterion_generation_id` 即 hold 住 close(`unsatisfied_not_witnessed`),每次 amend 换代后必须重新见证一次红。而封条在验证行动完成后**无诚实红可产**(证据已在、指纹已配),agent 被机器逼着在自己写的字段上表演状态转换——seq 7「新判据默认失败见证已记录」、seq 10「修正版判据默认失败见证已记录;**恢复真实 E2E passed 状态**并执行最终验证」,两次 amend 各仪式化一次。

## 关键账本事实(与票面互补)

- **机器计价链在 A1 上全部正确开火**:proof `gap`(`criterion_provenance` + `criterion_input_coverage`)→ acceptance(seq 11,`granted_by: self`)→ `provisional` → floor 抬 substantial(`proof_gap_acceptance`)→ 强制 fresh_context 评审(seq 12,reviewer `codex-field-option-review`,accepted,2 条 advisory)→ 终态 `achieved` + `proof_provisional: true`(seq 13)。**机制层没有一环失职。**
- **但评审看不见要评的东西**:收据(`docs/test-runs/2026-07-13-field-option-team-id-stability-e2e.json`)未提交、无版本史,翻转仪式对评审者不可见。强制评审开火了,却是对着一份可任意改写且改写不留痕的文件。
- acceptance 的 `granted_by ∈ {user, self}`(`task-engine.mjs:1034`),两个值机器都验不了真伪——「人」在 runtime 本体论边界外(16 决议 5 已裁)。

## 决议

### 决议 1:拆焊——绑定半边是真判据,verdict 字段是声明,判据禁读

封条的**绑定半边**(证据存在性、schema/env、**源码指纹比对**)是**真判据**:它是观测,且有自己诚实的红色词汇表——**缺据**(收据尚不存在)、**陈旧**(上代任务残留)、**漂移**(验完之后源码变了→指纹红,这是封条能产的最有价值的真红)。

**裁决半边**(`status` 等 agent 可写的 verdict 字段)**不是判据,是声明——判据禁止读它**。收据里的 verdict 是写给人和评审看的,不进退出码。

红见证的供给侧归**机器半边**(测试等真判据组件);收据永远不作红色杠杆。

### 决议 2:裁决通道 = proof_gap acceptance——归位,不新造

verdict 的正确通道**本来就存在**:proof gap acceptance 就是为「判据够不到的残余由谁站台」造的位置。A1 的病是把 verdict 塞进 JSON 让判据读(错误通道),acceptance 反而成了自签手续(正确通道空壳化)。

归位后:acceptance **如实署名**——人亲验则 `user`(声明,按 16 决议 4 记录不闸门),agent 一次性验证则 `self`;**self 合法且已被正确计价**(floor→substantial→强制评审,A1 全链开火)。**不需要 judgmentloop 的人终结动词**——那是品味型交付物的形态;这类活有客观真相,账 A 的第二双眼(强制评审)+ 诚实记账即够。内核不试图锚「人」(16 决议 5 边界)。

### 决议 3:指纹保留;收据必须入 git

指纹是封条里**唯一的观测**,保留,且是正解——它防「验完之后又改」。覆盖缺口(依赖/配置/数据/外部系统)由 `criterion_input_coverage != full` 如实计价——**已有机制,不扩**。

新增一条纪律:**收据必须提交进 git**(repo provenance)。买到的是**篡改可见性**(翻转仪式在版本史里无处藏身,强制评审第一次有东西可审),不买作者真实性(git 作者同样 agent 可写,不越「人」的边界)。A1 的收据未提交正是仪式得以不可见的原因。

### 决议 4:零新机制——缺口全在形态层,三件执行项进 #04

`provisional`/acceptance/floor/强制评审是**正确机制且已全链开火**;封条不是 `provisional` 换名字,`provisional` 是对封条的**如实标注**(taskloop 这一点从来做对了)。缺的是形态纪律:

1. **封条形态准入规则**(08 的形状,开单时就可判):开单问「业务真相机器今晚够得着吗?」——够不着→显式选封条形态:机器半边判据 + 绑定检查 + 收据计划 + **预告 acceptance 路径**。05 决议 2「计划必须自带可执行判据」由此在这类活上落地:可执行判据 = 机器半边+绑定,够不到的残余**开单时如实预告**而非关单时才发现。落点:workloop/planning skill 文档。
2. **判据写作三禁**:判据不读 agent 可写的 verdict 字段;红见证供给侧归机器半边;收据绑定用指纹不用字段。落点:loop-core 判据纪律。
3. **收据入 git**:封条类任务的证据文件必须提交,评审收据连同其历史。落点:planning/evidence 规范。

## 限制与翻盘条件

- **红见证逐代强索的残余压力**:amend 后若机器半边也无诚实红可产,压力仍在。随票 10 决议 4(`.mjs` 文件判据)大幅消退——A1 三次 amend 全是 PowerShell 载体病。残余情形记为限制,不立机制。
- **形态纪律不可强制**(本决议的失败模式):agent 仍可焊接+自批。缓解已在机器里——`criterion_subject` grant 照旧抬地板、强制评审照旧开火、票 13 落地后 untracked 写入进证据书。
- **最接近替代方案**:typed `human_verification` 事件入内核(一等公民化)。**翻盘条件**:宿主 payload 出现人的身份锚(地图雾区「人-面」graduate 条件——票 14 就此在该雾区挂了一个等着的消费者),或封条类任务频发且形态纪律成文后伪造仍复发——届时一等化重问。
- **边际收益停止点**:机读 lint(「判据读 untracked/agent 可写 verdict 字段」告警)今天不做——证据书 + 强制评审已覆盖观测面,第二道同功能门等复发证据。
