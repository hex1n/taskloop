## Blocking

### B1 — drive-relative Windows paths (`C:foo`) still evaded the control-state guard

`retired supervision runtime:29` (`WINDOWS_ABSOLUTE_PATH`), consumed at `readShellValue`

Round one settled on `/^(?:[A-Za-z]:[\\/]|\\\\[^\\/])/`, requiring a separator after the drive letter. Windows also honors *drive-relative* paths with no separator: `C:foo` means "foo relative to the current directory on drive C". Such a word fell through to POSIX escaping and was mangled exactly as the original bug mangled absolute paths. Reviewer reproduced:

```
targetOf('echo x > C:.git\config')  ->  [ 'C:.gitconfig' ]   // backslash eaten
controlPlaneWriteFailure(...)       ->  false                // no deny
```

The premise justifying the whole fix — a Windows-native executor resolves the final path — makes `C:.git\config` and `C:..\repoA\.git\config` live bypasses of the same guard. For an agent working in-repo the drive's current directory *is* the repo, so this resolves onto the real control files.

**Resolution**: dropped the separator requirement — `/^(?:[A-Za-z]:|\\\\[^\\/])/`. Covered by "a drive-relative Windows path cannot smuggle a control-state write past the guard" in `tests/foreign-session-scope.test.mjs`, which asserts denies for `C:.git\config`, `C:.workloop\task.json`, and `C:..\repo\.git\config`, and re-asserts that ordinary POSIX escaping (`/tmp/two\ words.txt`) is untouched.

### B2 — naming states were routed into the destructive archive path

`lib/application.mjs` — `incompatibleArtifactPath` / `SNAPSHOT_OWNED_AUTHORITY_FAILURES`

`loadV3Authority` also throws `ARTIFACT_NAME_CONFLICT` (both `events.jsonl` and legacy `events-v3.jsonl` present) and `ARTIFACT_NAME_MIGRATION_REQUIRED` (only the legacy name present — the state *every* taskloop-era upgrade lands in). Neither means any artifact is damaged; both are naming states whose sanctioned repair is `migrate-artifact-names --granted-by user`. Round one's dispatch had only two buckets, so both fell through to "archive the event store". Reviewer reproduced:

- **Dual-name conflict**: archived `.workloop/events.jsonl` — the current, authoritative log — leaving the legacy one. If the two ever diverge (the reason the loader refuses to choose an authority at all), this silently discards the newer history.
- **Migration required**: archived a perfectly healthy `task.json`; `status` afterward still reported the same migration requirement. A healthy artifact destroyed for no progress.

**Resolution**: added a third bucket. `ARTIFACT_NAME_FAILURES` now produces a refusal naming `migrate-artifact-names` instead of a source path, and `cmdArchive` returns that as an error. Covered by "archive-incompatible-state refuses naming states instead of retiring a healthy artifact", which exercises both codes and asserts the snapshot stays byte-identical and neither log is retired.
