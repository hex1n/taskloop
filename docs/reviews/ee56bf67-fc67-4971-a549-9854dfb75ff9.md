## Blocking

### B1 — the second-colon exclusion reopened the bypass for NTFS alternate data streams

`lib/supervision.mjs` — `windowsRootedPath`, drive branch

Round three added `return !raw.includes(":", 2)` so that sed-style text (`s:/old:/new:`) would keep POSIX escaping. But a legitimate Windows path can contain a second colon: an alternate-data-stream reference, `C:\repo\.git\config:hidden`. That is standard `path:stream` syntax honored by the Win32 API, `cmd`, and PowerShell, and the referenced bytes live in the **same file entry** as `config` — writing the stream is writing `.git\config`. The second colon disqualified it, so the word fell back to POSIX escaping and was mangled exactly as the original bug mangled everything else. Reviewer reproduced:

```
echo x > C:\repo\.git\config:hidden
  parsed target -> "C:repo.gitconfig:hidden"     (backslashes eaten)
  controlPlaneWriteFailure(...) -> null           (NOT denied)
cp secret.txt C:\repo\.git\config:hidden
  parsed target -> []                             (target lost entirely)
  controlPlaneWriteFailure(...) -> null           (NOT denied)
baseline, same file without the suffix -> denied
```

**Resolution**: replaced the exclusion with a colon count — `raw.split(":").length <= 3`. A drive path carries at most one stream suffix (two colons); three or more is sed syntax. Covered by "an alternate-data-stream suffix does not smuggle a write past the control-state guard", which asserts denies for both control artifacts via redirection and via `cp`, and re-asserts that the three-colon sed form still keeps POSIX escaping.

## Advisory

### A1 — a single-segment rooted path (`\.git`) does not match the rooted branch

The rooted branch requires a later separator (so a POSIX escape of a command word like `\ls` is not misread as a path), which excludes a control directory sitting directly at a drive root. Reviewer confirmed `echo x > \.git` parses to `.git` rather than `\.git`, but spent significant effort and **could not** turn it into an observable false allow: `canonicalWriteTarget`'s fallback resolves the mangled literal against `repo`, which lands on one of the tracked roots anyway. Recorded as a latent inconsistency with the function's documented contract, not a demonstrated bypass.

**Resolution**: not changed. Left as a known gap so the next round does not rediscover it as novel.

### A2 — the reducer-rejection path was verified only by hand, never by an assertion

Reviewer independently reconstructed both `EVENT_STORE_AUDIT_FAILED` (a structurally valid record the reducer refuses) and `MIXED_OR_INVALID_AUTHORITY` (a self-consistent but cursor-mismatched snapshot), and confirmed both route correctly through `incompatibleArtifactPath` — round three's claim holds. But no test covered them: both "poisoned record" tests trigger `CORRUPT_EVENT_AUTHORITY` at the parse layer, since `readEventStore` rejects an unknown `kind` before the reducer sees it. That left the exact failure mode this feature exists for verified twice by manual reproduction and zero times by the suite.

**Resolution**: accepted. Added "a structurally valid record the reducer rejects routes archival to the event log", which appends a duplicate `task_opened` — parses cleanly, reducer refuses — and asserts the log is archived while the snapshot survives byte-identical.
